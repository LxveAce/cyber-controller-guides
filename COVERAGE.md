# Guide coverage & backlog

Tracks which of Cyber Controller's shipped flash **profiles** have a written hardware guide, and seeds the ones that
don't with the authoritative facts needed to write them. The published guides themselves are listed in
[README.md](README.md) (auto-generated by `tools/build_readme.py`); this file is the **inverse** — what's still
missing — so contributors know where to help.

- **Shipped profiles:** 33 (`cyber-controller/src/config/profiles/*.json` — the SSOT).
- **Published guides:** 30 (see the README tables).
- **Profiles without a guide yet:** 7 (below). Software-OS / detector meta-guides (Arch, Kali USB, Tails, Jammer
  Detection) are additional guides beyond the profile set.

> The facts below are pulled straight from each profile's JSON (backend / board / chip / upstream repo) and are safe to
> rely on. A full guide is more than facts — the **buy → build → flash → integrate → troubleshoot** walkthrough must be
> **verified against the real device** before publishing (the repo's `verify:` discipline). Those steps are marked TODO.
> Several of these are offensive/regulated (deauth, BadUSB) — their guides must lead with the **authorized-use / your-own-
> gear** frame, same as the existing ⚠ entries.

## How to add a guide
1. Write `guides/<name>.md` following an existing guide's shape (what to buy → build → flash → **Cyber Controller
   profile** → troubleshooting), citing the exact `src/config/profiles/<id>.json` facts. Mark unverified steps `verify:`.
2. Add the entry to the guide index **data**, then regenerate: `python tools/build_readme.py`; build the PDF:
   `python tools/build_pdfs.py`.
3. Device-level walkthroughs benefit from the CC session's hands-on hardware knowledge — coordinate on
   `command-center/COORDINATION.md` before claiming one so two sessions don't write the same guide.
